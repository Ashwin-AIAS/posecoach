PoseCoach has 5 distinct layers. Each lives in its own folder. Click any tab to drill into it.

Top-level project structure
  • app/All Python backend code — FastAPI, YOLO inference, RAG chatbot, auth, database models
  • src/All React frontend code — camera, skeleton overlay, UI components, WebSocket hook
  • tests/Pytest tests for backend — angle logic, WebSocket behaviour, auth flows, history API
  • e2e/Playwright end-to-end tests — simulates a real user in a browser (camera → form score)
  • nginx/NGINX config — HTTPS termination, WebSocket proxy, rate limiting
  • deploy/Production Docker Compose — separate from dev, adds workers, SSL, watchtower
  • alembic/Database migration scripts — creates and updates your PostgreSQL schema safely
  • scripts/Utility scripts — test YOLO locally, evaluate model mAP accuracy, ingest RAG documents
  • docker-compose.ymlDev environment — starts FastAPI + PostgreSQL + Redis with a single command
  • .env.exampleTemplate for your secrets — copy to .env and fill in your keys (never commit .env)
💡 The app/ and src/ split is the most important boundary. Everything in app/ runs on the server; everything in src/ runs in the user's browser.
app/ — Python backend
Entry point app/main.py + core files ▶
  • app/main.pyThe FastAPI app itself. Registers all routers, loads YOLO model on startup, configures CORS and security headers, exposes /health and /health/deep endpoints
  • app/db.pyDatabase connection — async SQLAlchemy engine connected to PostgreSQL. All DB sessions come from here
  • app/models.pySQLAlchemy table definitions — Users and WorkoutSessions. This is your database schema in Python
  • app/cache.pyRedis helper — get/set/delete wrappers. Used for WebSocket session state when running multiple workers
  • app/metrics.pyPrometheus counters and histograms — tracks frames received, dropped, inference time, form score distribution
  • app/logging_config.pyStructured JSON logging setup. In production every log line is JSON; in dev it's coloured console output
Core feature app/inference/ — YOLO pose pipeline ▶
  • websocket.pyThe /ws/analyze WebSocket endpoint. Receives JPEG frames from the phone, puts them in a queue, returns keypoints + form score. Drops frames when the queue is full so the server never gets overwhelmed
  • predictor.pyRuns YOLO26-Pose on a single frame. Returns 17 body keypoints. Handles "no person detected" and "multiple people" edge cases. Runs in a thread so it doesn't block the async event loop
  • smoother.pySmooths keypoint positions across 5 frames using a rolling average — stops the skeleton from jittering on screen even when YOLO detection fluctuates slightly
These 3 files are your real-time pipeline: frame in → YOLO → smooth → score → back to phone. If you're debugging latency, start here.
Core feature app/analysis/ — form scoring logic ▶
  • angle_calculator.pyMaths — takes 3 keypoint coordinates (e.g. hip, knee, ankle) and returns the angle in degrees. This is the foundation of all form scoring
  • form_scorer.pyOne scoring function per exercise (squat, deadlift, curl, bench, OHP, lunge, plank). Each compares your joint angles against ideal ranges and returns a 0–100 score + coaching cues like "knees caving inward"
  • rep_counter.pyTracks the up/down phase of a movement by watching one primary angle over time. Counts a rep when the angle completes a full cycle (e.g. knee angle goes down then back up for a squat)
If form scores feel wrong, the bug is almost always in angle_calculator.py or the ANGLE_RANGES dict at the top of form_scorer.py — those are your tuning knobs.
AI chatbot app/chatbot/ — RAG coaching chatbot ▶
  • router.pyThe /chat/stream endpoint. Receives the user's question + their current session data (exercise, reps, form score). Streams the response back token-by-token using Server-Sent Events
  • rag.pyRetrieval step — searches ChromaDB for the 5 most relevant chunks from your exercise knowledge base that are relevant to the user's question
  • llm.pyGemini API integration — builds the prompt (context + session data + question), calls gemini-2.0-flash, streams tokens back. If Gemini is down, returns a hardcoded fallback message instead of an error
  • ingest.pyOne-time setup script — reads the 8 knowledge text files, splits them into chunks, embeds them, and stores them in ChromaDB. Run this once before the chatbot can answer anything
  • knowledge/8 plain text files (squat_technique.txt etc.) — this is the only source of truth the chatbot can use. It cannot answer anything not in these files
Auth & history app/auth/ + app/history/ ▶
  • auth/router.py/auth/register and /auth/login endpoints. Login returns a JWT stored in an httpOnly cookie (not localStorage — safer against XSS attacks)
  • auth/middleware.pyget_current_user — a reusable FastAPI dependency. Any endpoint that needs authentication just imports this. Returns the user or raises a 401
  • history/router.pyPOST /history saves a workout session; GET /history retrieves past sessions; GET /history/stats calculates your average form score trend per exercise over time
Middleware app/middleware/ ▶
  • security_headers.pyAdds security headers to every response — tells browsers not to render the app in an iframe, not to sniff content types, etc.
  • request_id.pyGenerates a unique ID for every request and attaches it to logs. When something breaks, you can search logs by request ID to find the full trace
  • timing.pyLogs a warning if any request takes more than 500ms. Helps you spot performance regressions before users notice
src/ — React frontend (runs in the browser)
Hooks src/hooks/ — the brains of the app ▶
  • useCamera.tsAccesses the phone camera (rear-facing by default). Handles permission errors with specific messages. Stops the camera when the phone screen locks and restarts it when the user comes back
  • useWebSocket.tsConnects to the backend WebSocket. Captures frames from the camera and sends them. Reduces quality automatically when connection is slow. Reconnects automatically with exponential backoff if disconnected
If something is wrong with the camera or the real-time connection, the bug is in one of these two hooks — they're the data entry point for the whole app.
Components src/components/ — what the user sees ▶
  • SkeletonOverlay.tsxDraws the green/orange/red skeleton lines on top of the camera feed using a canvas element. Colour reflects form score. Shows nothing if no person is detected
  • FormHUD.tsxThe heads-up display — score gauge (top right), rep counter (top left), coaching cues sliding in from the bottom, connection status badge (Live / Reconnecting / Offline)
  • ChatDrawer.tsxThe AI coaching chatbot drawer. Uses Server-Sent Events to stream tokens as they arrive. Shows "GymVision Coach is typing..." while waiting. Sends session context automatically with every question
  • ExerciseSelector.tsxThe chip bar for selecting which exercise you're doing (Squat, Deadlift, etc.). This tells the backend which scoring function to use
  • HistoryDrawer.tsxShows past workout sessions and a line chart of your form score trend per exercise over time (built with Recharts)
  • PermissionScreen.tsxShown when camera access fails — gives specific instructions for each error type (NotAllowedError = you denied permission, NotFoundError = no camera found, NotReadableError = camera in use by another app)
Context & config src/context/ + config files ▶
  • context/AuthContext.tsxStores login state globally. JWT is kept in memory (not localStorage) to reduce XSS risk. All components that need to know "is the user logged in" read from here
  • serviceWorker.tsPWA service worker — caches static assets so the app loads even with poor signal. Network-first for API calls so you always get live data when online
  • vite.config.tsBuild config — sets up the PWA plugin, TypeScript paths, and dev proxy (forwards /api requests to FastAPI during local development)
  • public/manifest.jsonMakes the app installable on Android/iOS home screens — defines the app name, icon, orientation (portrait), and theme colour
Infrastructure files
Docker docker-compose.yml (dev) ▶
  • fastapi serviceYour Python backend running on port 8000. Hot-reloads when you change code
  • postgres:15Your database. Data is stored in a named volume so it survives docker-compose restarts
  • redis:7-alpineFast in-memory cache. Stores WebSocket session data so multiple FastAPI workers can share state
docker-compose up starts all three. That's all you need in development — you don't need to install Postgres or Redis locally.
Docker deploy/docker-compose.prod.yml ▶
  • fastapi (2 workers)Runs with Gunicorn + 2 Uvicorn workers so it can handle concurrent WebSocket connections
  • nginx serviceSits in front of FastAPI. Handles HTTPS, redirects HTTP to HTTPS, proxies WebSocket connections, rate-limits 30 req/min per IP
  • watchtowerAutomatically pulls and restarts containers when you push a new Docker image — automated zero-downtime deploys
Database alembic/ — database migrations ▶
  • alembic/env.pyConnects Alembic to your async SQLAlchemy engine. Don't touch this unless you're changing the DB connection
  • versions/0001_initial_tables.pyCreates the Users and WorkoutSessions tables. Run with: alembic upgrade head
  • versions/0002_add_workout_sessions.pyAdds the workout history table with index on (user_id, exercise, created_at) for fast filtering
Always run alembic upgrade head before starting the server after pulling new code. The CI/CD pipeline does this automatically before every deploy.
CI/CD .github/workflows/ ▶
  • ci.ymlRuns on every pull request — pytest, vitest, docker build. Blocks the PR merge if any step fails
  • deploy.ymlRuns on push to main — runs all tests first, then runs Alembic migrations, then deploys frontend to Vercel and backend to Render, then checks /health/deep to confirm the deploy succeeded
Scripts scripts/ ▶
  • test_yolo.pyLoads YOLO26, runs it on a webcam frame, prints the 17 keypoints as JSON. Use this to confirm the model is working before touching any other code
  • eval_yolo.pyRuns the model on 50 test frames and calculates mAP@0.5 accuracy. Exits with code 1 if accuracy drops below 0.80 — acts as a quality gate when upgrading YOLO versions
Test files and what they check
Pytest (backend) tests/ ▶
  • conftest.pyShared test fixtures — creates an in-memory SQLite database (not your real Postgres), a mock YOLO model that returns fake keypoints, and a sample JPEG image for testing
  • test_angle_calculator.pyChecks angle maths — a correct squat angle gives score ≥ 85, a bad angle triggers the right cue, and edge cases (3 collinear points) don't crash the calculator
  • test_websocket.pyChecks WebSocket behaviour — valid frame returns all expected fields, blank image returns no_person_detected, sending 10 frames very fast causes at least one frame_dropped
  • test_auth.pyChecks auth — register then login works, duplicate email is rejected, invalid JWT returns 401, the 11th login attempt in 1 minute returns 429
  • test_history.pyChecks history — save a session then retrieve it, check that stats trend calculation is correct after 3 sessions
Vitest (frontend) src/__tests__/ ▶
  • useCamera.test.tsChecks the camera hook — mocks the browser camera API, tests that switching camera toggles facing mode, tests that locking the screen stops the stream and unlocking restarts it
  • useWebSocket.test.tsChecks the WebSocket hook — mocks the WebSocket connection, verifies that a high round-trip time triggers quality reduction, verifies exponential backoff timing on reconnect
Playwright (E2E) e2e/ ▶
  • camera_flow.spec.tsOpens the app in a real browser, grants camera permission, checks the skeleton overlay canvas is present, selects Deadlift and checks the chip is highlighted
  • chat_flow.spec.tsTypes a question in the chat drawer, checks that tokens stream in one by one (not all at once), checks that the "typing" indicator appears and then disappears
Real-time pose analysis — frame journey
Phone camera → useCamera.ts → useWebSocket.ts → WebSocket (JPEG frame)
websocket.py → predictor.py (YOLO26) → smoother.py → form_scorer.py
rep_counter.py → JSON response → SkeletonOverlay.tsx + FormHUD.tsx
Each frame completes this full round trip in under 80ms on CPU. The phone sends up to 15 frames per second.
RAG chatbot — question journey
User types question → ChatDrawer.tsx → GET /chat/stream (SSE)
rag.py → ChromaDB → top 5 chunks → llm.py → Gemini → SSE tokens stream
ChatDrawer.tsx renders tokens
What each env var controls
  • POSTGRES_URLWhere your database lives — used by db.py for all SQL queries and by Alembic for migrations
  • REDIS_URLWhere Redis lives — used by cache.py for WebSocket session state
  • GEMINI_API_KEYYour Google AI Studio key — used only by llm.py. Get it free at aistudio.google.com
  • JWT_SECRETSecret used to sign login tokens — must be a long random string, never shared
  • MODEL_PATHPath to yolo26n-pose.pt — if this is wrong, the server fails fast with a clear error on startup
  • ALLOWED_ORIGINSComma-separated list of frontend URLs that can talk to the backend — prevents other websites from calling your API
  • CHROMA_PATHWhere ChromaDB stores the embedded exercise knowledge — defaults to ./data/chromadb
  • SENTRY_DSNYour Sentry error tracking URL — if missing, error tracking is silently disabled (no crash)
